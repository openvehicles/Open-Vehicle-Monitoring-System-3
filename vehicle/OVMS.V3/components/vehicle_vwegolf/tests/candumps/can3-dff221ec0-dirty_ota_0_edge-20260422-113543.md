## Analysis (2026-04-22)

### Summary

Clima did not start. TX bytes are byte-for-byte identical to the known-working
capture `kcan-can3-clima_on_off.crtd`, so this is **not a BAP protocol
regression**. The clima ECU simply did not respond.

### Evidence

| Metric | This capture (113543) | Working capture (kcan-can3-clima_on_off) |
|---|---:|---:|
| Total frames on 0x17332510 (ECU push/ack) | 4 | 934 |
| op=4 ACK frames within 2s of each TX burst | 0 / 2 bursts | 1 / 1 burst (every time) |
| Exact `80 0a 49 59 {ctr\|0x80} 04 46 00` match | 0 | 100% |

Our TX bursts (start ctr=0x04, stop ctr=0x05) were clean:
`80 08 29 59 {ctr} 06 00 01` / `c0 06 00 96 00` / `29 58 00 {01|00}`
(frame 1 of start burst is missing from the log — logger dropped 8687/24208
messages — not a TX failure; frame 2+3 captured and stop burst captured in
full with matching sequence).

### Hypothesis: NM ring membership

Bus was nominally alive (5A7 TX flowing, 0x17330301 RX, 0x1B000067 RX), but
clima ECU was inert toward node 0x67. Recent commits that reworked the NM
path (review items 12–14: deferred BAP burst, 5A7 rate-limit, NM linger)
could have disturbed ring state from the ECU's perspective even though TX
itself looks right.

### Deferred — next diagnostic

Compare NM-ring traffic and `0x5EA`/`0x35C` power-state indicators between
`kcan-can3-clima_on_off.crtd` and this capture to confirm whether ECU saw
node 0x67 as a valid ring peer at command time. Rule in/out before touching
BAP payload further.

### Sequence actually run

This was **not** cap-18 (deferred path). Bus was already awake at capture
start (5A7 heartbeat and BAP ACKs active from t=0), so OVMS correctly
skipped the wake/NM-join TX. Effectively a direct-path (cap-19) run that
still failed despite bus-awake preconditions — strengthens the NM ring
hypothesis over the wake path.
